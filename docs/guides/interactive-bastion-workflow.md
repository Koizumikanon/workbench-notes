# 交互式堡垒机工作流

> 最后更新：2026-07-10 | 类型：运维指南
>
> 关键词：`ssh`、`bastion`、`terminal`、`automation`
>
> 经验标签：`FT`

某些堡垒机以终端菜单提供资产搜索和进入功能，而不是标准的 SSH TCP 跳板。这种模式应保留审计和权限边界，不能强行当作透明 `ProxyJump` 使用。

## 优先级

1. 能使用标准 SSH、`ProxyJump` 或受控 VPN 时，优先使用标准链路。
2. 必须经过交互式菜单时，使用人工选择或受控自动化模拟搜索与进入。
3. 不绕过堡垒机资产授权、审计或审批机制。

## 人工流程

```text
登录堡垒机
-> 等待资产列表
-> 搜索目标关键字
-> 核对目标名称、协议和用户
-> 进入目标机器
-> 执行获批准的操作
```

## 自动化边界

适合自动化：资产列表读取、关键词搜索、目标选择、只读命令、会话状态记录。

需要额外确认：写入 `authorized_keys`、修改网络策略、创建账号、重启服务、执行批量命令、保存真实资产缓存。

自动化程序应把真实地址、用户、私钥、known_hosts 和资产缓存放在本地私有配置；代码仓库只保留模板、接口和脱敏示例。

## 面向网站的设计

网页不应在 HTTP 请求中直接运行长 SSH 会话。推荐结构：

```text
Web UI -> API -> Job Queue -> Worker -> Bastion/SSH -> Result Store -> Web UI
```

每个任务记录目标、授权人、执行步骤、输出脱敏结果、失败原因和回滚入口。
