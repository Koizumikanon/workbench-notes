# 文档站的分类与扩展

> 最后更新：2026-07-10 | 类型：文档架构指南
>
> 关键词：`mkdocs`、`navigation`、`taxonomy`、`documentation`

## 当前能力

MkDocs 适合把文档按文件夹组织，再在 `mkdocs.yml` 中定义清晰导航：

```text
docs/
├─ command-recipes/  # 高频命令
├─ guides/           # 方法和原则
├─ ai-cli/           # AI 命令行工具
├─ ai-skills/        # 可复用 AI 工作流
└─ cases/            # 脱敏工作经验案例
```

可以继续增加二级、三级目录、专题索引、全文搜索、代码复制和独立页面。目录深度本身不是瓶颈，关键是导航和命名是否仍易于查找。

## 推荐分类

| 内容 | 放置方式 |
| --- | --- |
| 单工具常用命令 | `command-recipes/` |
| 多步骤交付流程 | `command-recipes/` 或 `cases/` |
| 原理、规范、决策方法 | `guides/` |
| AI CLI、skills、MCP、plugins | `ai-cli/`、`ai-skills/` |
| 已验证的匿名工作经验 | `cases/` |

每页保持“最后更新、类型、关键词”。新页面先搜索现有内容，能补充就不要新建近似页面。

## 何时做结构评审

出现以下信号时，AI 应在项目或阶段完成后建议一次结构评审：

- 新增了第三个以上的同类页面，已有导航开始难找。
- 同一主题出现重复命令、重复案例或不同术语。
- 页面数量超过约 50，且需要按专题、技能、场景或经验标签交叉浏览。
- 需要版本文档、多语言、自动生成 API 文档、评论、权限或多人编辑。
- 需要交互式表单、数据库、任务状态、登录权限或实时数据。

## 升级路线

先增强现有站点，再考虑迁移：

1. 增加专题索引、关键词约定和更细的导航。
2. 引入 MkDocs 的标签、博客、重定向或文档生成插件。
3. 需要复杂前端交互时，评估 VitePress、Docusaurus 或独立 Web 应用。
4. 需要私有访问控制时，使用带鉴权的部署，不把敏感内容放进公开静态站。

不要仅因为文件数量增加就迁移。只有当前工具无法满足明确需求时，才先提出迁移方案、成本和回滚计划。
